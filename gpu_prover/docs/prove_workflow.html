GPU prove() Workflow

GKR + WHIR proof generation pipeline — stage-by-stage data flow

1. High-Level Pipeline

The prove() function (proof.rs:304) orchestrates 6 GPU-compute stages interleaved with small host callbacks for Fiat-Shamir transcript hashing. Everything is enqueued asynchronously onto the GPU exec_stream; the host returns a ProofJob handle immediately.

2. Stage 1 — Witness Generation, LDE, and Merkle Commitment

The prover's opening move: commit to the witness before seeing any challenges. Raw execution trace is processed into committed polynomial evaluations. Everything in this stage runs on GPU except the tiny tree-cap readback.

3. Stage 2 — Forward Pass (GKR Dimension Reduction)

The GKR protocol's core computation. The circuit has multiple layers. At each layer, all constraints are evaluated on GPU and the polynomial domain is halved. After all layers, a final dimension reduction collapses the 4 output types (PermutationProduct, Lookup16Bits, LookupTimestamps, GenericLookup) down to small polynomials that are read back to host.

4. Stage 3 — Backward Pass (Sumcheck Protocol)

"Was the forward dimension reduction done correctly?" The backward pass proves it via sumcheck, layer by layer top→base. Each layer's sumcheck has multiple rounds, each consisting of a GPU reduction kernel followed by a tiny host callback (commit univariate coefficients to transcript, draw next challenge).

5. Stage 4 — Base Layer Claims

The backward pass ended with: "the base-layer polynomials should evaluate to X at random point r." This stage computes those evaluations on GPU from the committed trace data (still on device from Stage 1). It uses the equality polynomial eq(r,x) and batch reductions.

6. Stage 5 — WHIR / FRI Folding (Polynomial Commitment Proof)

The prover committed to polynomials in Stage 1 and claims they evaluate to specific values (Stage 4) at a random point (Stage 3). WHIR proves this. First, all 3 trace holders are batched into a single polynomial. Then, iterative sumcheck-style folding rounds halve the degree, producing intermediate Merkle-committed "oracles." Finally, query proofs open the committed polynomials at random positions.

8. Async Scheduling Model

H2D transfers for setup/trace/decoder data are scheduled before prove(), running on the h2d_stream while the host prepares. Inside prove(), the exec_stream awaits the "transferred" event, then all stages are enqueued. Host callbacks (green stripes) fire inline on the exec_stream for transcript hashing.

Seed lineage: initial_seed (from caps hash) → draw 3 lookup challenges → forward pass uses them → commit explicit evaluations → draw eval_point + batching_challenge → snapshot into backward_shared_state → backward pass advances seed per sumcheck round → WHIR reads seed (advances by drawing WHIR batching challenge) → WHIR draws per-round challenges from seed.
